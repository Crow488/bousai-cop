# セキュリティ設計メモ

公開にあたり、2つのAI（Claude / Codex）で独立に監査を行い、指摘を反映した記録。
防災情報を扱うため、「動くこと」より「間違った安心を与えないこと」を優先している。

## 1. 外部データの取り扱い（XSS対策）

本アプリは気象庁・国土地理院のAPIから取得したデータを画面に描画する。
データ源は政府の公式APIだが、**「信頼できる情報源だから」を理由にサニタイズを省かない**方針とし、
外部由来の文字列（地震名・天気文・発表官署名・避難所名・住所・ブラウザのエラーメッセージ等）は
すべて `esc()` を通してからHTMLへ埋め込む（`js/app.js`）。

### Leaflet の bindPopup と CVE-2025-69993 について

Leaflet 1.9.4 以下の `bindPopup()` に関する CVE-2025-69993（XSS）が登録されているが、
Leaflet 開発チームはこれを**仕様どおりの動作であり脆弱性ではない**としている
（`bindPopup()` は仕様上HTML文字列を受け取るメソッドであり、CWE-79 の責任は
未サニタイズの入力を渡す呼び出し側にある、という立場。修正版はリリースされていない）。

参照: https://github.com/Leaflet/Leaflet/issues/10214

本アプリはこの見解に沿って、**`bindPopup()` へ渡すすべての動的値をエスケープ済み**にすることで対処している。

## 2. プライバシー

- 利用者データを保存する**サーバーを持たない**。アクセス解析・広告タグも入れていない。
- 位置情報は、標高判定のため国土地理院APIへ、ハザード判定のため地図タイル要求（区画番号）として
  国土地理院・気象庁へ送信される。ブラウザから各機関へ直接アクセスするため、各機関にはIPアドレス等が伝わる。
- 指定した位置はブラウザの localStorage にのみ保存し、外部へ送信しない。
- 以上を「このサイトについて」ダイアログで開示している。

## 3. 公開APIへの負荷配慮

- 更新間隔は5〜30分に制限。
- **タブが非表示の間はリクエストを送らない**（開きっぱなしのタブが無駄な通信を続けるのを防ぐ）。
  表示に戻った時点で即座に再取得して鮮度を回復する。

## 4. フェイルセーフ（防災ツール固有の設計）

通信失敗を「異常なし」と誤報しないことを最優先している。

- ハザードタイルが **404（＝その地点に公表データなし）** の場合のみ「区域外」と表示する。
- **通信失敗・タイムアウトの場合は「判定できず」** と表示し、「区域外」「平常」と断定しない。
- 「区域外」の表示には常に「公表データ上の判定であり、想定外の災害は起こりうる」旨を併記する。
- 色の照合には許容幅を設けているが、**地図の背景色を「浸水想定あり」と誤判定しない**ことを
  テストで検証している（`test/logic.test.js`）。実際にこのテストで、白が「浸水想定0.5〜3m」に
  誤分類されるバグを発見・修正した。

## 5. リポジトリの衛生

- 内部向けの作業文書・AI委任時の生ログは公開対象から除外し、`.gitignore` で誤コミットを防ぐ。
- 公開前に、コミット履歴全体に対して機微情報のスキャンを実施している。

## 6. 既知の限界

`README.md` の「既知の限界」を参照。特に、**本アプリは公式の防災情報そのものではなく参考情報**であり、
避難の判断は市町の避難情報・気象庁の発表に従うべきものである。
